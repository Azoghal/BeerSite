Moving on from python, in search of a low level engine for 2d games, I found Love2D. Love2D was built for use with lua, so I had to learn lua, which proved pretty simple due to the likenesses with python. To begin with, I made some simple (and pretty rubbish) space invaders/ top down shooter games.

Simple space invaders style game (Love2d w/ lua)

Simple top down shooter game (Love2d w/ lua)

Although the first game is impossible to win, this does mean that all players have to listen to my wonderful stock audio integration when they die. Love2d provides basic libraries to implement file I/O, including a seperate and quite extensive music library. This meant that it was trivial to implement audio, but it was a useful experience as I had to look through the documentation and being adept at trawling through documentation has since proven invaluable. The top down game did not really build upon the space invader-alike, however it did leave me feeling that I wasn't doing something right - I was using an array of identical objects, with identical behaviours, but with different starting conditions. This rudimentary system poorly mimicked object orientation, which was pointed out by friend who was a more experienced programmer than I was. He introduced me to the concept of object orientation, and showed me the middleclass library for object orientation in lua.

Third game was a single player pong-alike

I decided to make a Tower Defence game as an introduction to object orientation, as it allowed good use of the concepts of OOP (Like inheritance ; Level of specific turret : specific turret : basic turret). I did not utilise OOP to its full potential, for example I rewrote a lot of methods that could have been inherited from the base class. I implemented a system for the enemies to navigate a hard coded map. This was not ideal, but was suitable as the game had only one map, and if I increased the number of maps, they would not be random, so my approach could be expanded. Basically the algorithm required an input for the x and y locations of corners, and would work out a vector to get from a to b, then re-evaluate to find the vector for b to c, and so on. This had some issues as the position was often not exactly the same as the destination position. To solve this problem I changed the check from being equal, to being further than the destination. This did not impact too hugely on the accuracy of the path following unless the game was running at a very low tick-rate, which did not tend to be a problem due to the simple graphics and few objects. Unfortunately I lost all but the start of the code, but still have some assets that I ended up using for the next TD Game.
